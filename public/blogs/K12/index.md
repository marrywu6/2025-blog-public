### 一、 产品愿景与核心逻辑

**产品名称（暂定）：** 智考AI (SmartExam AI)
**核心理念：** 数据自动化生产（AIGC for Data）+ 学习个性化服务（AI Agent for Learning）。
**关键差异点：** 彻底摒弃传统题库公司依靠成百上千人手动录题、打标签的模式，完全利用OCR、CV（计算机视觉）和LLM（大语言模型）构建自动化的数据工厂。

### 二、 “零人工录入”的数据工厂（核心壁垒）

这是本项目能否成功的关键。你需要构建一个**自动化ETL流水线**，将互联网上公开的PDF真题试卷转化为结构化的数据库。

#### 1. 数据源获取 (Crawler)

*   **自动爬虫：** 针对全国各省市教育局官网、大型试卷分享网站，自动抓取中考、高考真题及模拟题的PDF/Word文件。

#### 2. 试卷版面分析 (Document Layout Analysis)

*   **技术点：** 使用基于深度学习的视觉模型（如LayoutLMv3, YOLOv8）识别试卷结构。
*   **任务：** 自动区分“大标题”、“题干”、“选项”、“图片”、“解答区域”。
*   **切题：** 将整张试卷自动切割成一道道独立的题目图片。

#### 3. 高精度OCR与公式还原

*   **技术点：** 必须支持复杂的数学公式识别（LaTeX格式）。
*   **推荐模型：** Nougat (Meta开源), Pix2Text, 或商业API（如Mathpix，但成本高，建议自训微调PaddleOCR）。
*   **输出：** 将图片题目转化为机器可读的Markdown + LaTeX文本。

#### 4. AI自动做题与打标 (Auto-Tagging)

*   **解析生成：** 将识别后的题目文本输入给强推理模型（如GPT-4o, Claude 3.5 Sonnet, 或微调后的Qwen-Math）。
    *   *Prompt:* “请做这道题，并给出标准答案、详细的分步解析。”
*   **知识点打标：** 让LLM根据教育部大纲，自动给题目打上标签。
    *   *标签维度：* 学科（数学）、章节（三角函数）、考点（正弦定理）、难度（困难）、题型（选择题）。
*   **数据入库：** 将题目文本、向量Embedding、AI生成的解析、标签存入数据库。

### 三、 应用端功能模块设计

#### 1. 超级OCR搜题 (Snap & Solve)

*   **功能：** 用户拍照 -> 识别文本 -> 向量数据库检索（Vector Search）。
*   **逻辑：**
    *   如果库里有原题：直接返回真题的标准解析和考点。
    *   如果库里无原题：调用LLM即时生成解答，并自动将这道新题存入题库（实现题库的自我生长）。

#### 2. AI 知识点讲解 (AI Tutor Agent)

*   **交互：** 类似ChatGPT的对话界面，但被限制在教学上下文中。
*   **苏格拉底式教学：** 当用户问“这题怎么做”时，AI不直接给答案，而是问：“你觉得第一步应该先分析哪个条件？”引导学生思考。
*   **关联微课：** AI自动生成该知识点的简短文字总结或语音讲解。

#### 3. 智能出题与变式训练 (AI Question Generation)

*   **同题变式：** 针对用户做错的真题，利用LLM进行**同构变式**。
    *   *指令：* “保持这道题的逻辑和考点不变，修改数值和背景故事，生成一道新题。”
*   **针对性出题：** “我想练习‘导数在单调性中的应用’，请给我出5道高考难度的题。” -> AI从库中检索或生成。

#### 4. 智能错题本 (Smart Mistake Notebook)

*   **自动化整理：** 用户搜过的题、做错的题自动加入。
*   **举一反三：** 错题本不仅记录错题，还会在错题下方自动推荐3道相似考点的题目供复习。
*   **遗忘曲线推送：** 根据艾宾浩斯遗忘曲线，自动提醒用户复习某道错题。

### 六、 潜在风险与应对

1.  **准确性风险（最大痛点）：**
    *   *问题：* OCR识别错误（如 $x^2$ 识别为 $x2$）会导致整题全错；LLM解题可能逻辑对但计算错。
    *   *方案：* 引入**Verifier（验证者）机制**。让两个不同的模型分别做一遍，答案一致才通过；或者利用Python脚本进行符号计算验证。

2.  **版权风险：**
    *   *问题：* 它是中高考真题，通常属于公共资源，但直接抓取商业题库（如XX题库）的数据会有法律风险。
    *   *方案：* 严格限制数据源为公开的政府/学校官网PDF，通过“加工”（OCR重排、AI解析）转化数据所有权。

3.  **算力成本：**
    *   *问题：* 每次搜题都跑一遍GPT-4成本太高。
    *   *方案：* “缓存机制”。常见的题目一旦被解决过一次，下次直接调库，不调大模型。只有新题才调用大模型。

### 七、 总结建议

这个项目的核心不是APP开发，而是**后台的数据处理能力**。
建议先不要急着做APP，先写Python脚本，跑通 **[PDF -> OCR -> LLM Cleaning -> 结构化JSON]** 这个流程。如果你能证明你能以极低成本自动生产高质量题库，这个应用就已经成功了一半。


完全可以！利用现成熟的**API服务（SaaS）**和**低代码/无代码平台**进行“乐高积木式”的组装，可以极大地降低开发门槛，甚至一个人就能在几周内跑通核心流程。

这种方式的核心不再是“研发模型”，而是“**编排（Orchestration）**”。

### 二、 具体实现方案：如何用工具串联

我们将系统分为两个部分：**后台（自动化造数据）** 和 **前台（给学生用）**。

#### 1. 后台：自动化“题库工厂”流水线

**目标：** 把网上下载的PDF试卷，变成数据库里一道道打好标签的题。

*   **Step 1: 抓取试卷 (Apify)**
    *   使用 Apify Store 里的通用爬虫（Web Scraper），配置目标网站（如某些公开的真题分享站）。
    *   自动下载 PDF 文件到云存储（如 AWS S3 或 阿里云OSS）。
*   **Step 2: 切割与识别 (Mathpix API)**
    *   编写一个简单的 Python 脚本（或使用 Zapier/Make 自动化工具）：
    *   调用 **Mathpix API** 将 PDF 转为 Markdown。Mathpix 能够自动识别版面，把题目切分开。
*   **Step 3: 智能清洗与打标 (DeepSeek/GPT API)**
    *   将 Mathpix 识别出的文本发送给 **DeepSeek**。
    *   *Prompt:* “你是一个数据整理专家。这是OCR识别的文本，请修正其中的乱码，提取出题目、选项，并根据中国高考大纲生成知识点标签（如：数学-导数-单调性-困难）。以JSON格式输出。”
*   **Step 4: 入库 (Supabase)**
    *   将清洗后的 JSON 数据和题目文本的 Embeddings（向量）存入 **Supabase** 数据库。

#### 2. 前台：App 功能实现

**目标：** 实现搜题、讲题、错题本。

我们使用 **Dify** 或 **Coze** 来搭建后端逻辑，前端用 **FlutterFlow**。

*   **场景 A：拍照搜题**
    1.  **前端 (FlutterFlow):** 用户拍照 -> 上传图片。
    2.  **中台 (Dify Workflow):**
        *   节点1：调用 **SimpleTex/Mathpix** 识别图片，拿到 LaTeX 文本。
        *   节点2：调用 **向量检索 (Vector Search)** 在 Supabase 里查找相似度 > 0.95 的题目。
        *   分支判断：
            *   *若找到：* 直接返回库里的标准解析。
            *   *若没找到：* 进入“AI解题模式”。
    3.  **AI解题模式 (DeepSeek):** 将题目文本发给模型，要求生成详细步骤。同时触发“入库”动作，把这道新题存入数据库（实现题库自动增长）。

*   **场景 B：AI 讲解与追问**
    *   利用 **Coze** 创建一个 Bot。
    *   配置“知识库”插件，连接你的真题数据库。
    *   提示词设置：“你是一名苏格拉底式的中学老师，不要直接给答案，要一步步引导学生...”

*   **场景 C：错题变式训练**
    *   用户点击“举一反三”。
    *   **Dify** 调用 LLM：“基于这道题的考点（{tags}）和逻辑，生成 3 道类似的各种变式题，数值不同但解法类似。”
    *   返回给前端显示。

### 三、 成本与可行性分析

#### 1. 优势

*   **速度快：** 传统开发需要半年，用这一套方案，MVP（最小可行性产品）可能只需要 **2-3周**。
*   **无需算法工程师：** 不用自己训练OCR模型，不用维护复杂的RAG架构，全是调接口。
*   **数据质量高：** Mathpix 的识别能力远超绝大多数自研模型。

#### 2. 劣势与成本（关键！）

*   **API 成本（主要瓶颈）：**
    *   **Mathpix：** 价格较贵（约 $0.002 - $0.01 / 页）。如果我们要处理 10,000 套试卷（每套 10 页），成本可能达到几千美元。
    *   *解决建议：* 使用国产的 **SimpleTex** 或 **PaddleOCR (百度AI开放平台)** 作为平替，虽然精度略低，但成本大幅下降。
*   **依赖性：** 如果 Mathpix 挂了或者涨价，你的核心流程会受影响。

### 总结

使用 **Mathpix/SimpleTex (OCR)** + **DeepSeek (Brain)** + **Dify/Coze (Flow)** + **Supabase (Data)** 是目前实现“纯AI驱动、零人工录入”教育应用的最快、最稳健的路径。

你不需要招聘几十人的团队，**1个全栈开发者 + 1个产品经理** 就能把这个东西跑起来。



如果要实现**“接近零成本”**（Zero Marginal Cost），我们就不能依赖按次收费的商业API（如Mathpix或GPT-4）。你需要用**开源模型**替代商业模型，用**免费额度**替代付费服务，用**本地算力**替代云端算力。

这意味着你需要“**用技术换钱**”。核心思路是：**本地部署（Local Host）** 或 **薅羊毛（Free Tier）**。

以下是为你定制的三套“极限低成本”方案：

---

### 方案一：超级“羊毛”流（推荐：利用 Google Gemini Flash）

**核心逻辑：** Google 的 **Gemini 1.5 Flash** 模型目前提供**免费**的 API 额度（限制速率，但做题库整理足够了）。最关键的是：它是**原生多模态**，它**不需要OCR**！它可以直接看懂图片里的数学公式并输出 LaTeX。

*   **成本：** 0元。
*   **架构：**
    *   **OCR + 意图理解 + 解题：** 全部交给 **Gemini 1.5 Flash (Free Tier)**。
    *   **数据库：** **Supabase**（免费版足够存几万道题）或 **MongoDB Atlas**（免费版）。
    *   **后端运行：** **Render** 或 **Vercel**（免费托管 Python/Node 代码）。

*   **实施流程：**
    1.  **抓取：** 用 Python 脚本下载 PDF。
    2.  **切图：** 用 Python (`PyMuPDF`) 把 PDF 切割成一张张图片（按题目切或整页切）。
    3.  **调用 Gemini：**
        *   把图片传给 API。
        *   *Prompt:* "请识别这张图片中的所有数学题目，提取题目文本（公式用LaTeX格式），并给出详细解析和知识点标签。以JSON格式返回。"
    4.  **入库：** 拿到的 JSON 直接存入 Supabase。

*   **优点：** 开发极其简单，跳过了最难的 OCR 环节。
*   **缺点：** 免费版 API 有速率限制（每分钟请求数 RPM 限制），大批量跑数据需要排队慢慢跑。

---

### 方案二：硬核“本地开源”流（适合有显卡/游戏本）

**核心逻辑：** 如果你有一台带 NVIDIA 显卡的电脑（如 RTX 3060/4090），你可以把所有模型下载到本地跑。完全离线，**完全免费**，不用看任何云厂商脸色。

*   **成本：** 0元（只需电费）。
*   **架构：**
    *   **OCR (开源替代)：** **GOT-OCR2.0** 或 **PaddleOCR**。
        *   *推荐：* **GOT-OCR2.0** 是近期开源的强力模型，对通用场景和公式支持较好。或者使用 **Umi-OCR** (基于Paddle的开源封装工具)，支持批量离线识别。
    *   **大模型 (本地推理)：** **Ollama** + **Qwen-2.5-Math**。
        *   *解释：* 阿里开源的 **Qwen-2.5-Math** 是目前也是全球最强的开源数学模型之一。通过 Ollama 工具一键在本地运行。
    *   **向量数据库：** **ChromaDB** (本地 Python 库，不需要服务器)。

*   **实施流程：**
    1.  **本地环境：** 安装 Python, Ollama, PyTorch。
    2.  **OCR 脚本：** 编写 Python 脚本，调用本地的 GOT-OCR 模型，把 PDF 图片转成 Text/LaTeX。
    3.  **AI 清洗：** 脚本自动把识别出的文本发给本地运行的 Ollama (Qwen-Math)，让它修正错别字、生成解析、打标签。
    4.  **存文件：** 结果直接保存为本地 JSON 文件或 SQLite 数据库。

*   **优点：** 数据隐私极好，无任何调用次数限制，想跑多少跑多少。
*   **缺点：** 需要硬件支持（至少 8GB-16GB 显存的显卡），前期配置环境比较折腾。

---

### 方案三：混合“白嫖”流（GitHub 开源项目 + 免费 API）

**核心逻辑：** 拼凑目前市面上所有提供“免费额度”的最优解。

| 模块            | 零成本/低成本替代品    | 说明                                                         |
| :-------------- | :--------------------- | :----------------------------------------------------------- |
| **OCR**         | **Umi-OCR** (软件)     | GitHub 高星项目，基于 PaddleOCR，完全离线免费，支持批量导入图片导出文字，勉强能识别公式（精度不如 Mathpix）。 |
